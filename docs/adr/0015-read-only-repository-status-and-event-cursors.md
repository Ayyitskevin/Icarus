# ADR 0015: Read-only repository status and event cursors

- Status: Accepted
- Date: 2026-07-20
- Extends: [ADR 0014](0014-loopback-api-react-workspace.md)
- Related: [ADR 0002](0002-sqlite-event-history.md), [ADR 0010](0010-inherited-opencode-workflow-security-hold.md)

## Context

The first M3 workspace slice exposes a safe persisted snapshot and supports
manual refresh. The next bounded slice needs enough live observation to answer
whether a selected repository has drifted and whether a selected run has new
events. Returning raw Git output, dirty paths, event payloads, or independently
timed run/history reads would widen the disclosure boundary or present a torn
story. A streaming transport, filesystem watcher, new persistence model, or
browser action authority is unnecessary for this observation-only step.

## Decision

Add project-scoped, sanitized repository observation. Each response presents
independent availability, worktree, HEAD, branch, and configured-base-relation
fields. Missing repositories, identity mismatch, unresolved refs, and observation
errors remain explicit in their relevant fields and never masquerade as a clean
worktree. Detached HEAD is represented as `branch: null`, while worktree
cleanliness remains independently truthful. Dirty filenames, dirty-file counts,
file content, repository paths, and raw Git output are intentionally omitted.
Observation uses only fixed read-only Git controller operations with network,
hooks, external programs, prompts, and optional index locks disabled. It is
point-in-time, nonpersistent state: it does not update project/run rows or append
events.

Add a read-only event metadata endpoint for one selected run. Pages are ordered
by event sequence, begin strictly after the supplied sequence cursor, and use a
fixed service-owned maximum size. Each item contains only sequence, type, a
host-controlled label, timestamp, and a fixed host-generated `evidenceSection`.
The endpoint never returns or derives display text from `payload_json`.
Separately, the full run presenter reads the run row, approvals, and a fixed
200-event metadata tail from one coherent SQLite read snapshot. The append-only
sequence high-water mark is its event cursor and timeline total. Browser action
state is derived only from that bounded tail; the CLI keeps complete
payload-bearing history. An incomplete tail starts with unknown action state; a
retained self-establishing transition may make it known, but an ambiguous suffix
stays `unknown` with CLI guidance rather than being guessed.

The workspace may short-poll only the currently selected run while the document
is visible. It permits one current request, pauses when visibility is lost,
aborts work on selection change or unmount, and applies bounded backoff after
failures before returning to the short steady-state interval after success. A
selection/request revision guard rejects late or out-of-order responses. Live
full-run responses are accepted only when their event cursor is at least the
newest event revision already observed from the separate metadata endpoint. Live
updates link only to a closed set of evidence sections whose fragment identifiers
are generated by Icarus; repository, provider, event, and check text never becomes
an HTML identifier or navigation target.

This slice adds no Server-Sent Events, WebSocket, filesystem watcher, schema
migration, runtime dependency, background daemon, browser approval, mutation,
execution, arbitrary-command, commit, push, or deployment authority. It leaves
the guarded CLI lifecycle and the ADR 0010 release hold unchanged.

## Consequences

The workspace gains bounded, review-only freshness without another control plane
or persistent source of truth. Repository observation is advisory and
point-in-time; a later guarded action must still perform its own authoritative
identity, HEAD, base-ref, and cleanliness checks.

Short polling is deliberately simpler than a streaming lifecycle but must be
tested for hidden-document pause, abort, backoff, cursor ordering, coherent
full-run reads, the cross-request event-cursor guard, and stale-response
rejection. Richer file/status, diff, and history navigation, including dirty
filenames or counts and event payload presentation, remains deferred. Any
streaming transport, persistence/schema change, new dependency, or browser
authority requires a later explicit design and acceptance evidence.

The implementation fails closed when effective repository, included, worktree,
or private-cache config defines clean/smudge/process filters,
`core.alternateRefsCommand`, or configured `hook.*.command` programs. The
controller also disables the `post-checkout` hook event at command scope;
local-file transport is the only permitted Git transport and lazy object fetching
is disabled. The config preflight and Git command remain separate subprocesses,
so concurrent hostile same-user config mutation is a documented non-goal rather
than an isolation claim.
